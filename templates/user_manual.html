Manifest User Manual

Comprehensive guide to the e-signature platform for the DockLabs suite.

FAQ

Overview

Manifest is the e-signature platform for the DockLabs suite. It captures multi-party signatures on PDFs through configurable, ordered workflows with full audit trails — and serves as the cross-product signing handoff layer for the rest of the suite.

Reusable flows

Build a signing workflow once; instantiate it as many packets as you need.

Multi-party signing

Sequential, ordered signers. Each signer's turn unlocks when the previous step is signed.

ESIGN / UETA compliance

Cryptographic verification, IP capture, immutable audit trail, FOIA-exportable.

Suite handoff layer

Peer products send packets here for signature; Manifest returns the signed PDF on completion.

Roles

Manifest's signer roles can be sourced two ways:

  • Suite mode — when deployed alongside other DockLabs products, roles come from KeelUser.Role via OIDC SSO claims (product_access).
  • Standalone mode — roles are managed in Manifest's UI at /roles/. Each role has a key, label, optional description, and an is_active flag.

Staff-only surfaces (creating flows, editing placements, initiating and cancelling packets, viewing audit trails) require is_staff plus the grant_manager or equivalent agency-staff role. Any authenticated user can sign their own assigned step.

Getting Started

Signing in
1 From any DockLabs product, click Manifest in the fleet switcher, or visit https://manifest.docklabs.ai/.
2 Click Sign in with DockLabs (the suite OIDC button).
3 You'll land on /dashboard/ — the Packets list.
If you arrived via an email signing link (/sign/<step_id>/), you'll be sent through the same login flow and bounced back to the signing page once authenticated.

Flows (templates)

A SignatureFlow is the reusable template that drives every packet: a name, an optional description, an ordered list of steps, the document templates that will be signed, and the signature placements on those documents.

/flows/ lists every active flow. Click into one to manage:

  • Steps — the ordered signing sequence.
  • Documents — the PDF templates attached to the flow.
  • Placements — where each signer signs on each page.

Flows can be linked to a GrantProgram when deployed alongside Harbor, or stand alone.

Creating a flow

1Click Flows → New flow.
2Provide a name and optional description.
3Add steps, upload one or more PDFs, and define placements for each signer.
4Save. The flow is now ready to instantiate as packets.

Steps & Signers

A SignatureFlowStep is one signer in the flow. Each step carries:

  • Order — execution sequence (1 = first signer).
  • Label — e.g. "Program Officer Approval", "Director Sign-off".
  • Assignment type — user (a specific person) or role (a role key like director).
  • Required — whether the step blocks completion.

Steps execute sequentially. Step 2 only becomes ACTIVE after step 1 is SIGNED. The packet completes when every required step is signed.

Edit at /flows/<flow_id>/steps/create/ and /steps/<id>/edit/ (staff only).

Documents & Placements

A SignatureDocument is the PDF template attached to a flow. Files are validated by Keel's FileSecurityValidator (extension allowlist, size limit, malware scan). Page count is auto-populated on upload.

Placements

A SignaturePlacement says "this field, on this page, at this position, belongs to this signer." Coordinates are stored as percentages of page dimensions, so placements stay correct regardless of rendering DPI or zoom level.

Field typePurpose
SignatureA captured signature image.
InitialsA short initials capture.
DateThe auto-stamped signing date.
Printed nameThe signer's typed name.

Placement editor

/documents/<id>/placements/ opens a visual placement editor — a PDF preview with drag-to-position fields. Each placement is bound to a specific step (signer). Saves go through the JSON API at /api/documents/<id>/placements/.

Roles Registry

/roles/ (staff only) is the standalone-mode role manager. Use it when Manifest is deployed alone and you can't rely on SSO-issued role claims.

Each role has:

  • Key — machine-readable slug (e.g. director).
  • Label — human-readable name.
  • Description — optional context.
  • Active flag — toggle without deleting.

When a step's assignment type is role, the matching role key is resolved at packet-initiation time to whichever user holds that role.

Template Builder Wizard

/builder/ (or /builder/<flow_id>/ to edit) is a unified wizard that walks staff through creating a flow end-to-end:

1Flow metadata (name, description).
2Steps + signer assignments.
3Document upload.
4Placement editor.
5Save.

The wizard saves through /api/builder/save/ — a single round-trip that creates or updates the flow plus its dependents atomically.

Packets (active signing sessions)

A SigningPacket is an instance of a flow being executed. Listed at /packets/ (also mounted at /dashboard/). Each packet carries a flow, a title, an optional source entity (generic FK to any model — Harbor Award, Bounty Opportunity, etc.), a status, the final signed PDF, and timestamps for create / complete / cancel.

Initiating a packet

Visit /packets/initiate/<flow_id>/ (staff only). Pick the source entity (if any), confirm signer assignments, and submit.

Manifest creates a SigningStep for each flow step, sets step 1 to ACTIVE, sends the first signer their email link, and the packet enters IN_PROGRESS.

Cancelling a packet

/packets/<id>/cancel/ (staff only). Provide a reason; cancelled packets are terminal — signers can no longer sign their steps. Cancellation is recorded in the audit trail.

The Signing Experience

When you're the active signer, you receive:

  • An email with a /sign/<step_id>/ link.
  • A signature_required notification visible in your sidebar bell.
  • An item in your Helm dashboard's Awaiting Me column.

Click the link. Manifest renders the PDF with your placement fields highlighted, a signature capture pad (type / draw / upload), a signing intent confirmation, and Sign / Decline buttons.

Signing

Submit fires /sign/<step_id>/complete/. Manifest:

  1. Records the signature image, IP, and timestamp.
  2. Marks the step SIGNED.
  3. Activates the next step (if any) and emails its signer.
  4. If yours was the last step, generates the final signed PDF and marks the packet Completed.

Declining

/sign/<step_id>/decline/ — record a reason. The step transitions to DECLINED and the packet to Declined (terminal). Other signers stop receiving prompts.

Saved Signatures

/my/signatures/ is your library of saved signatures for fast reuse:

  • Typed — a font-rendered signature from your name.
  • Drawn — captured on a touch / mouse pad.
  • Uploaded — a PNG you supplied.

Each signature has a label (e.g. "Formal", "Initials") and an optional default flag. Defaults pre-populate the signing pad when you sign. /my/signatures/<id>/default/ toggles the default; /delete/ removes a saved signature.

Cross-Product Handoff

Manifest is the suite's signing handoff layer. The contract:

1Inbound — a peer product calls keel.signatures.client.send_to_manifest(). The peer creates a ManifestHandoff row pointing at the Manifest packet UUID.
2Manifest signs — the packet runs through its normal sequential signing sequence.
3Outbound completion — Manifest POSTs the signed PDF to the source product's webhook signed with MANIFEST_WEBHOOK_SECRET. The source's packet_approved handler attaches the signed PDF, advances its workflow status, notifies collaborators, and registers the file with keel.foia.export.
Standalone mode works fully. A flow with no source_entity GFK runs the same way; the signed PDF is available at /packets/<id>/. Peer products work when Manifest is unavailable too — their "Send to Manifest" controls fall back to a local "mark approved + upload signed PDF" path.

When a packet is initiated by a peer call, the Packet detail page shows the source entity (with a clickable URL back to the source product when configured).

Helm Inbox Feed

Manifest exposes two helm-feed endpoints:

EndpointPurpose
/api/v1/helm-feed/Aggregate metrics for Helm's "Across the suite" dashboard (active packets, completed in period, declined, etc.).
/api/v1/helm-feed/inbox/?user_sub=<sub>Per-user inbox: the SigningSteps where this user is the active signer and the parent packet is IN_PROGRESS, plus that user's unread Manifest notifications.

Both require Authorization: Bearer $HELM_FEED_API_KEY (skipped in DEMO_MODE). The per-user endpoint resolves the OIDC sub claim to the local user via SocialAccount(provider='keel', uid=sub). Unknown subs return an empty items[] (200), not 404.

Audit Trail

Every meaningful event is written to a Keel AuditLog row, immutable and FOIA-exportable:

  • Flow created / edited / deleted.
  • Document uploaded / deleted.
  • Placement created / moved / deleted.
  • Packet initiated / cancelled.
  • Step activated / signed / declined / reminded.
  • Signed PDF generated.

/packets/<id>/audit/ (staff only) renders the per-packet trail with timestamps, actor, IP, and structured payload. The trail is what gives ESIGN / UETA-compliant evidence of consent and process.
